# Optional Capabilities Decision Log

| ID | Date | Decision | Rationale | Consequence |
| --- | --- | --- | --- | --- |
| D-001 | 2026-04-19 | Govern optional capabilities work under `exec-ctrl` | The initiative spans architecture, optional dependencies, UI, persistence, and validation | Work must follow control record, audit log, and decision log before being considered complete |
| D-002 | 2026-04-19 | Treat optional functionality as capability packs | This keeps the base app lightweight and makes optional behavior explicit | Backend and UI should implement installable, enableable, testable packs rather than hard-coded feature branches |
| D-003 | 2026-04-19 | ChromaDB is a provider, not the architecture | Prevents overbinding the design to a single optional dependency | Build abstraction and local provider first, then add Chroma behind the same interface |
| D-004 | 2026-04-19 | Add a dedicated Capabilities workspace instead of relying on Settings alone | Operational systems like indexing, health, and source management need more room than a modal | Capability installs begin in Settings, but ongoing management belongs in a first-class workspace surface |
| D-005 | 2026-04-19 | Implement `splunk_deeplink_tools` before visualization and export packs | Deeplinks are lower risk and immediately useful in investigation workflows | First enhancement-pack delivery should prioritize operator actionability before richer artifact generation |
| D-006 | 2026-04-19 | Land backend capability APIs before the dedicated workspace UI | The framework needed stable persistence and control contracts before UI binding | Phase 3 now exposes capability state, install, enable, disable, test, and config APIs that the future workspace can consume |
| D-007 | 2026-04-19 | Keep capability lifecycle management in the Capabilities workspace and treat chat RAG as a per-session usage toggle | Install/enable state is operational and persistent, while whether chat should use an installed optional RAG provider is session-specific | The chat settings copy now assumes an installed capability, and assistant responses must surface `capability_usage` evidence when optional RAG contributes context |
| D-008 | 2026-04-19 | Use a deterministic local hash-based embedding function for `rag_chromadb` | The optional Chroma provider must work without model downloads or external network dependencies | Chroma indexing remains installable as an optional package while retrieval stays local, deterministic, and compatible with offline/demo environments |
| D-009 | 2026-04-19 | Implement deeplink generation with a capability-local `web_base_url` override, MCP URL derivation fallback, and a dedicated build endpoint | The deeplink pack needed to work without changing the global MCP config schema while still giving operators an override when Splunk Web differs from the MCP endpoint | `splunk_deeplink_tools` can build search links from `/api/capabilities/deeplinks/build`, the capability stays optional, and chat/capability surfaces can request links without duplicating URL construction logic |
| D-010 | 2026-04-19 | Stage `export_tools` as a deterministic bundle generator instead of promising PDF/PPTX rendering in Phase 5 | Existing report artifacts and persona runbooks already provide stable source material, while direct PDF/PPTX generation would add format-specific dependencies and reduce determinism too early | `export_tools` now ships zip, manifest, and summary outputs first; richer export formats are deferred follow-on work rather than blockers for the optional-capabilities initiative |
| D-011 | 2026-04-19 | Keep capability lifecycle controls workspace-first for this initiative and defer mirrored Settings controls | The Capabilities workspace already satisfies the operator-control requirement, and duplicating lifecycle flows into Settings would expand scope without changing the core runtime model | Initiative completion is not blocked on Settings parity; if reopened later, mirrored controls should be treated as a focused UX enhancement |
| D-012 | 2026-04-19 | Limit enhancement-pack evidence requirements to implemented chat and mission flows for this initiative | Capability evidence is already visible in chat previews, deeplink actions, and mission export flows; expanding that pattern into every downstream artifact surface would be additive rather than foundational | The initiative can close with current evidence visibility in place, while broader artifact/discovery evidence propagation remains an explicit future follow-up |
