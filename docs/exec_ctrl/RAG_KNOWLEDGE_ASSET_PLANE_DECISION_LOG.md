# RAG Knowledge Asset Plane Decision Log

| ID | Date | Decision | Rationale | Consequence |
| --- | --- | --- | --- | --- |
| D-001 | 2026-04-20 | Treat introduced RAG inputs as managed knowledge assets under `rag_chromadb` | The existing optional capability framework already owns indexing, health, and operator controls | The first usable slice extends the current capability model instead of introducing a separate RAG subsystem |
| D-002 | 2026-04-20 | Put the management plane inside the existing `Capabilities` workspace | Asset lifecycle and indexing state are operational concerns, not transient chat-only settings | Operators manage import, preview, and deletion where capability state is already visible |
| D-003 | 2026-04-20 | Support pasted text and common text-based file uploads first | This captures the highest-value documentation and system-context cases without opening binary parsing scope prematurely | PDF and richer binary ingestion remain explicit follow-on work rather than hidden partial support |
| D-004 | 2026-04-20 | Auto-reindex imported assets only when indexed retrieval is already installed, enabled, and restart-ready | Operators should get immediate usability when the capability is live, but imports should still work before capability activation | Asset import is available early, while retrieval refresh remains explicit and predictable when the indexed provider is not yet live |
| D-005 | 2026-04-20 | Use deterministic enrichment and operator context briefs instead of LLM-generated ingestion summaries for the next slice | The initiative backlog called for richer summarization and context building, but injecting LLM summarization into import would widen scope, add model dependence, and reduce determinism | Managed assets now persist focus terms, key points, and usage guidance, and preview results explain why assets matched without making ingestion dependent on LLM availability |
| D-006 | 2026-04-20 | Self-heal derived managed-asset enrichment and trigger reindex before retrieval when stored asset sections drift from current deterministic output | Live browser validation exposed a real gap: asset cards could reflect refreshed enrichment while preview chunks still came from stale stored markdown and index content | Existing managed assets now refresh stored asset sections on load, and preview/search paths reindex before retrieval when asset timestamps outrun the last index build |
| D-007 | 2026-04-20 | Keep asset detail inspection inside a shared capability-workspace panel instead of introducing a separate asset page | The management plane already centralizes lifecycle state, preview output, and delete controls, so a second navigation surface would add complexity without improving the operator flow for this slice | Operators can open asset detail from either the asset catalog or preview matches and inspect stored sections plus chunk-browser output in one place |
| D-008 | 2026-04-20 | Stage PDF support as deterministic text extraction with explicit page markers instead of broad binary-document ingestion | The next bounded slice needed practical PDF uploads, but widening immediately into scanned-document OCR or broader office formats would add heavy dependencies and reduce determinism | Managed assets now accept `.pdf` uploads through `pypdf`, store extracted content under `PDF Page N` sections, and keep Word or broader binary ingestion as explicit follow-on work |
| D-009 | 2026-04-20 | Stage DOCX support as deterministic heading, paragraph, and simple-table extraction instead of broad office-document ingestion | The next bounded slice needed practical Word upload support, but widening immediately into legacy `.doc`, spreadsheet, or richer office parsing would add more dependencies and blur the operator evidence model | Managed assets now accept `.docx` uploads through `python-docx`, preserve visible headings plus simple table rows in stored sections, and keep broader office ingestion as explicit follow-on work |
| D-010 | 2026-04-20 | Add stable preview-to-chunk traceability instead of reopening ingestion scope after DOCX | The next bounded improvement needed clearer operator trust in preview evidence, while additional file-format work was explicitly deferred until current ingestion proved stable in real use | Managed-asset previews now carry stable chunk identifiers, matched assets expose traceable chunk summaries, and the shared detail panel highlights the chunk-browser sections that powered the current preview |
| D-011 | 2026-04-22 | Close the initiative after preview traceability validation rather than leave it implicitly active for backlog items | All must-pass success criteria are satisfied, and the remaining work is deferred follow-on scope rather than an open implementation gap in the managed knowledge-asset plane | `rag-asset` is now marked `completed`, and broader ingestion or governance work must reopen under a fresh controlled slice instead of extending this one informally |
